# Program Card block — plan & rationale

## Goal

Reproduce the "Cohort Configure and Manage Adobe Experience Platform" card from
the assignment screenshot as an AEM Edge Delivery Services block, twice:

1. **Static** — every field authored directly in the block table (like `Cards`).
2. **Dynamic** — same visual output, but the content is sourced from elsewhere
   instead of retyped.

Both must follow the Block Collection principles from `AGENTS.md` /
aem.live docs: intuitive to author, no hard dependencies, responsive,
context-aware, localizable (no hard-coded content), and accessible.

## Naming

Block name: **`program-card`**, not `cohort-card`. The screenshot is one
instance of a recurring pattern (courses, cohorts, certifications all use the
same visual shape), and the Block Collection guidance explicitly favors names
"general enough so it can be reused without having to change the underlying
content model." `program-card` covers all three without renaming later.

## Content model (static authoring)

Key/value rows, one field per row — the same shape as the `Hero` block, chosen
over `Cards`' positional-cell layout because this block renders **one**
instance per table, not a repeated list, so labeled rows are more
self-documenting for an author than positional columns:

| Program Card |  |
| --- | --- |
| Category | My Learning |
| Heading | Cohort Configure and Manage Adobe Experience Platform |
| Status | In Progress |
| Format | Cohort |
| Level | Professional |
| Duration | 4 weeks |
| Description | Join a four-week cohort experience focused on building practical skills in Adobe Journey Optimizer... |
| Image | (image) |
| Link | https://example.com/learning/cohort-aep |

Every row except Heading is optional — `parseFields()` builds a lookup and
every `build*()` helper skips cleanly if its field is absent, per AGENTS.md:
"authors may omit or add fields to a block, so your code must handle this
gracefully."

`Status` accepts arbitrary author text; it's slugified into a `data-status`
attribute so `In Progress` / `Completed` / `Not Started` get themed colors,
and anything else still renders with a sane neutral default instead of
breaking.

## Content model (dynamic authoring)

Considered three options and picked reference-driven fetching over a
query-index-json lookup or a "static fields + live-status-only" hybrid,
because this exact repo already has a proven, idiomatic precedent for it:
the `Fragment` block (`blocks/fragment/fragment.js`) fetches
`{path}.plain.html` and inlines another page's content. Reusing that same
mechanism means the dynamic block behaves the way an AEM developer reading
this codebase would already expect.

Authoring is deliberately tiny — a variant of the same block, named
`Program Card (dynamic)` in the table (EDS turns the parenthetical into a
`dynamic` modifier class: `class="program-card dynamic"`), with a single row
containing a link to the **source page**:

| Program Card (dynamic) |
| --- |
| /learning/cohort-aep |

The source page is expected to carry its own canonical **static**
`program-card` block. At render time the dynamic block:

1. Reads the linked path.
2. Fetches `{path}.plain.html`.
3. Finds the `.program-card` element inside that response.
4. Rewrites any `./media_*` relative image URLs to be relative to the source
   path (the same fix-up `fragment.js` does, for the same reason).
5. Swaps those raw rows into itself and runs them through the **exact same**
   `parseFields` / `renderCard` used for static authoring.

Net effect: the visual card is generated by identical code either way — only
step 0 (where the rows come from) differs. Editing the canonical page once
updates every dynamic card pointing at it, which is the actual point of a
"dynamic" block: single source of truth instead of copy-pasted content.

This is implemented as **one block folder** with a modifier class, not two
separate block folders, because that's how EDS itself models variants
(`Cards (carousel)`, `Columns (dark)`, etc.) — introducing a second
`program-card-dynamic` folder would just duplicate `renderCard` and the CSS
for no benefit.

## The creative/needed addition: loading + failure states

A static card is synchronous — it's on the page or it isn't. A dynamic card
depends on a network request, so it has two states a static card never needs:
*mid-flight* and *failed*. Leaving both unhandled (blank block) would violate
the same "handle it gracefully" principle the field-level code already
follows, so:

- **Loading**: while the fetch is in flight, the block gets an `is-loading`
  class that renders a CSS-only shimmer placeholder (no extra DOM, no JS
  animation loop) sized to roughly the card's final height, so the layout
  doesn't jump when content arrives. Respects `prefers-reduced-motion`.
- **Failure**: if the path is malformed, the fetch fails, or the source page
  has no `.program-card` block, `renderFallback()` renders a small card
  reusing the existing `.program-card-content` / `-heading` /
  `-description` classes (no new CSS needed) with a plain-language message
  and a link to the source page — so a broken reference degrades to
  "here's a link" instead of a silent hole in the page.

## Files

```
blocks/program-card/program-card.js    # decorate() — static parse, dynamic fetch, shared render
blocks/program-card/program-card.css   # scoped styles, mobile-first, 900px breakpoint
drafts/learning/cohort-aep.html        # canonical source page (static card) — pretty HTML
drafts/learning/cohort-aep.plain.html  # same page's body-only markup, what the dynamic block fetches
drafts/my-learning.html                # consumer page: dynamic card + a second static card for comparison
```

`cohort-aep.plain.html` exists only because the local `aem up --html-folder`
dev server doesn't auto-derive `.plain.html` from `.html` the way the real
aem.live backend does for authored content — it's a local testing fixture,
not something a real project would hand-author.

## Testing performed

- `npm run lint` (`eslint` + `stylelint`) on the new files — clean.
- `aem up --html-folder drafts --html-mount /` locally.
- `curl` the pretty page, the `.plain.html` fragment, and the consumer page —
  correct markup/status codes.
- Headless-Chrome render check of both `/learning/cohort-aep` (static) and
  `/my-learning` (dynamic card + a second static card side by side):
  correct layout at desktop width, correct status-badge coloring per variant
  (`in-progress` pink, `not-started` neutral), bullet-separated meta list,
  offset accent frame behind the image, no console errors, final DOM shows
  both blocks at `data-block-status="loaded"` with no leftover raw rows.
- Found and fixed one real bug during this pass: the authored `<p>` wrapping
  Heading/Category text was being copied verbatim into the `<h3>`/eyebrow,
  producing `<h3><p>…</p></h3>`. Added `contentNodes()` to unwrap a lone `<p>`
  wrapper before building those elements.
- Note: headless Chrome's plain `--screenshot` flag sometimes captures before
  the dynamic block's extra `fetch()` resolves (a tooling race, not an app
  bug) — `--dump-dom --virtual-time-budget=8000` was used to confirm final
  state reliably.
- `images/img.png` (552KB, unoptimized, untracked leftover in the working
  copy) was resized/recompressed to `images/program-card-sample.jpg` (~15KB)
  for use as the drafts' placeholder photo, per AGENTS.md's requirement that
  committed assets be optimized. It's a placeholder only — swap it for a
  real photo when authoring for real.
